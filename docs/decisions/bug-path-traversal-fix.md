# Path Traversal Vulnerability Fix

## Problem
The `SpeciesCatalogStore.Load` and `SpeciesCatalogStore.Save` methods accepted unvalidated path strings, allowing potential directory traversal attacks (`../`). If user-controlled data was passed to these methods, an attacker could read from or write to unexpected directories on the filesystem.

## Alternatives Considered
1. **Absolute Path Verification**: Ensure that the resulting path is rooted inside the configured local app data directory or working directory using `Path.GetFullPath`.
   * *Rejected*: `SpeciesCatalogStore` handles paths from different contexts (LocalAppData in production, `/tmp` directories in unit tests). Attempting to enforce a single base directory breaks legitimate use cases such as isolated temp catalogs generated by unit tests.
2. **Simple Contains("..") Check**: Ensure the input path does not have standard relative directory up-traversal characters.
   * *Selected*: Since the intended usage uses absolute path variables combined properly from the OS Environment variables (or deterministic temp folders), adding a naive `..` filter effectively blocks unexpected traversal by user inputs, avoiding false positives where valid temp folders are used.

## Solution and Rationale
We implemented a fast fail via `if (path.Contains("..", StringComparison.Ordinal))` in `SpeciesCatalogStore.Load` and `SpeciesCatalogStore.Save`, which throws an `ArgumentException`.
We included a new unit test class `SpeciesCatalogTraversalTests` with `Xunit` assertions to ensure paths with `..` fail predictably in both the load and save branches.
